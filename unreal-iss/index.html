Unreal 초기화 스테이트 시스템을 이용한 컴포넌트 초기화 개선

2025년 10월 21일
#UnrealEngine InitStateSystem

개요

Unreal 을 사용하고, 협업하면서 자주 겪었던 문제는 초기화 시점 문제이다. 단순하게 특정 컴포넌

트가 초기화 되기 위해서, 다른 컴포넌트나 액터 가 먼저 초기화 되어야 하는 그런 상황이 많다.

이런 직접적인 의존성을 줄이고, Character의 BeginPlay 안에서 AComponent→Init(),

BComponent→Init() …. 이런 수동적인 초기화 코드들을 더 좋은 방향으로 바꿀 수 있을지 고민했다.


1. 전통적인 초기화 방식의 한계

//  수동 순서 관리
void AMyCharacter::BeginPlay()
{
    Super::BeginPlay();
    
    HealthComponent->Initialize();
    InputComponent->Initialize();
}

// Tick에서 반복 체크
void UHealthComponent::TickComponent(float DeltaTime, ...)
{
		// 성능 문제는...
		// 언제 초기화 되는지 알수 있나??
    if (!bInitialized)
    {
        if (AbilitySystem != nullptr)
        {
            Initialize();
            bInitialized = true;
        }
    }
}

// Timer로 실행
void UHealthComponent::BeginPlay()
{
		// 이것도 마찬가지...
		// 타이머도 관리해줘야 하네??
    GetWorld()->GetTimerManager().SetTimer(
        TimerHandle, 
        this, 
        &UHealthComponent::TryInitialize, 
        0.1f
    );
}

위 코드와 유사한 초기화 방식을 안 짜본 사람은 없을 것이다. 간단한 소규모 게임에서는 정말

효과적인 코드가 될 수도 있다.


하지만 규모가 커질 수록 다음과 같은 문제가 발생할 수 있다.

개인적으로는 협업에서의 문제점이 가장 크다고 생각한다.

  • 타이밍 이슈 :
    • 컴포넌트에서 ASC 가 필요하다면? 지금 초기화 된건지? 어디서 볼 수 있지?
    • Health 와 InputComponent 의 순서가 중요하다면? 한 줄만 실수로 바꾸면 큰일…
    • 비동기 데이터 로딩 - CharacterDefinition 같은 Primary 에셋이 로딩 중일 수 있음
  • 멀티 플레이 이슈 :
    • 만약 멀티 플레이 상황에서는 PlayerState 가 지연 복사될 경우에는? 아예 컴포넌트들이 초기화되지 않을 수도 있는데?
  • 유지 보수 :
    • 컴포넌트가 추가되면 계속 수동 관리해줘야 한다.
    • 컴포넌트들의 Initialize 함수를 호출하는 것을 까먹으면? (이 경우도 생각보다 많다)
    • 초기화 순서가 바뀌면?
  • 협업 위험 :
    • 컴다른 개발자가 컴포넌트를 추가하려는데 순서를 모른다..
    • 이 컴포넌트는 초기화 시 뭐가 필요한지 알기 어렵다

4. 좀 더 개선된 ISS 사용하기 위한 고민

Lyra 의 ISS 는 좋은 시도였지만, 몇 가지 아쉬운 부분을 내 프로젝트에 적용해보려고 한다.


1. ISS 를 사용한다면, 모든 컴포넌트에 적용하기 (수동 초기화 로직 피하기)

  • 어딘 쓰고 어딘 안쓰고 같은 일 벌어지지 않기
  • 그 컴포넌트의 CanChangeState 와 HandleStageChange 함수만 보면 초기화 시 어떤 의존성이 필요한지, 어떤 초기화를 하는지 알 수 있게

2. 상태 이름을 직관적으로 변경하기

  • 언리얼에서는 초반 초기화 로직에 필요한 것들은 ASC, PlayerState, PlayerController 와 같은 것과 커스텀 데이터에 해당한다고 생각했기 때문.
  • 따라서, 상태 이름을 InitState_ASCInitialized 이런식으로 변경하기

3. 모든 컴포넌트들의 상태를 한 곳에서 조율하는 중앙 관리자 클래스 추가하기

  • PawnStateManager 의 상태가 현재 이 Pawn 의 상태라고 정의
  • PawnStateManager 만 보면 전체 초기화 순서와 진행 상황을 알 수 있도록 하기
  • PawnStateManager의 상태에 따라, 나머지 컴포넌트들이 대기하고 초기화하도록 한다.
bool UBSHealthComponent::CanChangeInitState(...) const
{
    if (CurrentState == InitState_Spawned &&
        DesiredState == InitState_ASCInitialized)
    {
        // PawnStateManager 야 나 ASC 필요해
        return Manager->HasFeatureReachedInitState(
            GetOwningActor(), 
            UBSPawnStateManagerComponent::NAME_PAWNSTATEMANAGERCOMPONENT,
            InitState_ASCInitialized
        );
    }
    return true;
}

void UBSHealthComponent::HandleChangeInitState(...) 
{
    if (CurrentState == InitState_ASCInitialized)
    {
        // ASC 준비됐구나! 나 초기화할게~~
        const ABSCharacter* Character = Cast<ABSCharacter>(GetOwningActor());
        InitializeWithAbilitySystem(Character->GetBSAbilitySystemComponent());
    }
}

3. 중앙 관리자 클래스 추가하기

중앙 관리자는 실제 리소스가 로드되었는지 체크한다.

그리고 다른 모든 컴포넌트가 각 상태에 도달했는지 확인하고, 다음 상태로 진입한다.

현재 Pawn 의 모든 컴포넌트가 어떤 상태에 있는지 디버깅하기도 쉽고, 한 곳에서 흐름을 볼 수 있다는 것이 장점이다. 

// ===== BSPawnStateManagerComponent.cpp =====
// ✅ 중앙 관리자가 모든 것을 조율
bool UBSPawnStateManagerComponent::CanChangeInitState(...) const
{
    const ABSPlayerState* PS = GetPlayerState<ABSPlayerState>();

    // === 1단계 진입: ASCInitialized ===
    if (CurrentState == InitState_Spawned &&
        DesiredState == InitState_ASCInitialized)
    {
        // 실제 리소스 체크: PlayerState와 ASC 확인
        if (PS && PS->GetBSAbilitySystemComponent()->HasBeenInitialized())
        {
            UE_LOG(LogBS, Log, TEXT("ASC 준비 완료!"));
            return true;
        }
        
        UE_LOG(LogBS, Log, TEXT("⏳ ASC 대기 중..."));
        return false;
    }

    // === 2단계 진입: CharacterDefinitionInitialized ===
    else if (CurrentState == InitState_ASCInitialized &&
             DesiredState == InitState_CharacterDefinitionInitialized)
    {
        // 조건 1: 모든 컴포넌트가 1단계 완료했나?
        if (!Manager->HaveAllFeaturesReachedInitState(
            GetOwningActor(), 
            InitState_ASCInitialized, 
            NAME_PAWNSTATEMANAGERCOMPONENT))
        {
            UE_LOG(LogBS, Warning, 
                TEXT("⏳ 일부 컴포넌트가 아직 ASCInitialized 도달 안함"));
            return false;
        }
        
        // 실제 리소스 체크: CharacterDefinition 로드 확인
        if (PS && PS->GetCharacterDefData())
        {
            UE_LOG(LogBS, Log, TEXT("CharacterDef 로드 완료!"));
            return true;
        }

        UE_LOG(LogBS, Warning, TEXT("⏳ CharacterDef 로딩 중..."));
        return false;
    }

    // ... 나머지 단계들
    return false;
}

실제로 위 코드를 실행했을 때의 로그이다.

Pawn의 모든 컴포넌트가 어떤 상태에 있는지 알 수 있다.



중앙 관리자도 HandleStateChange 함수로 각 상태에, 필요한 이벤트를 Broadcast 할 수 있다.

void UBSPawnStateManagerComponent::HandleChangeInitState(...)
{
    UE_LOG(LogBSInitState, Log, 
        TEXT("[Manager] State %s -> %s"), 
        *CurrentState.ToString(), 
        *DesiredState.ToString());
    
    // ASCInitialized 도달시 이벤트 발생
    if (CurrentState == InitState_ASCInitialized)
    {
        const ABSPlayerState* PS = GetPlayerState<ABSPlayerState>();
        if (PS && PS->GetBSAbilitySystemComponent())
        {
            // 📢 다른 컴포넌트들에게 알림
            OnAbilitySystemInitialized.Broadcast();
        }
    }
}
